## [19:15:00] 🤖 **지니**
태기! L/T Framework V3에서 **대용량 엑셀 다운로드**를 지원하기 위해 아키텍처적으로 설계된 핵심 메커니즘과 유틸리티 클래스는 다음과 같습니다.

### 1. SXSSFWorkbook 기반 메모리 스트리밍 (OOM 방지)
L/T Framework의 백엔드 공통 모듈에 구현된 `ExcelUtil.java` 및 `ExcelDrmUtil.java`는 Apache POI의 **SXSSF (Streaming Usability-focused XSSF)** 라이브러리를 핵심 엔진으로 사용합니다.
* **동작 원리:** `new SXSSFWorkbook(500)` 설정을 통해 메모리상에 최대 **500개의 행(Row)**만 유지하고, 임계치를 초과하는 데이터는 즉시 디스크 임시 파일(XML)로 플러시(Flush)하여 버퍼링합니다.
* **효과:** 수십만 건의 대량 데이터를 로드하고 셀을 생성할 때 발생할 수 있는 Java Heap Memory 오버플로우(`OutOfMemoryError`) 문제를 근본적으로 방지합니다.

### 2. 시트 분할 및 청크 단위 플러시 (`writeSheetChunk`)
엑셀 자체의 물리적인 한계(최대 1,048,576행) 및 성능 문제를 극복하기 위해 `writeSheetChunk()` 메서드가 구현되어 있습니다.
* **동작 방식:**
  - 시트당 데이터 임계치인 `sheetSizeNew = 1,000,000` (100만 행)을 기준으로 데이터가 이를 초과하면 자동으로 여러 개의 시트(`Sheet1`, `Sheet2`...)를 나누어 생성합니다.
  - 대량 데이터를 루프 돌며 쓰기 작업을 할 때, 주기적으로 `((SXSSFSheet) sheet).flushRows(list.size())` 처리를 호출하여 메모리를 비우고 디스크 파일로 저장합니다.

### 3. DRM 암호화 연계 지원 (`ExcelDrmUtil` & `DrmUtil`)
기업용 보안 요구사항(Fasoo DRM 등)에 대응하기 위해 일반 엑셀 추출 외에 **`ExcelDrmUtil.java`**가 추가로 탑재되어 있습니다.
* `SXSSFWorkbook`을 활용해 대용량 스트리밍 엑셀 임시 파일을 먼저 완성한 뒤,
* 다운로드 직전 `DrmUtil.java` 및 `FassoDrmHandler`를 호출하여 로그인한 사용자(`LoginUser`) 정보를 매핑한 DRM 암호화 파일(`encrypt`)로 실시간 포장하여 사용자 브라우저에 안전하게 전송합니다.

### 4. 프론트엔드-백엔드 연계 흐름
* **Front (Vue 3 / Vite):** `request.js`에 정의된 범용 `download` 헬퍼 함수가 Axios 요청 시 `responseType: 'blob'`을 지정해 서버로부터 이진 데이터 청크를 다운로드 스트림으로 받아옵니다. 다운로드가 완료되면 `file-saver`의 `saveAs`를 사용해 로컬 파일로 저장합니다.
* **Back (Spring Boot):** `@PostMapping` 컨트롤러 엔드포인트에서 `HttpServletResponse` 객체와 엔티티 리스트를 받아 `ExcelUtil<T>.exportExcel(response, list, sheetName)`을 실행하여 스트리밍 아웃풋 스트림으로 전송합니다.

### 동작 원리 분석
1. **프론트엔드 전면 대기 (Full-screen Loading):**
   - [request.js](file:///c:/Users/htg09/project/valuesplay/admin-ui/src/utils/request.js)의 `download` 함수가 실행되면 `Loading.service`를 통해 화면 전체를 반투명한 레이어로 덮는 **Full-screen Loading 스피너**가 동작합니다.
   - 메시지: `MSG_DATA_DWLD_WAIT` ("데이터 다운로드 중입니다. 잠시만 기다려주십시오.")
   - 백그라운드 색상: `rgba(0, 0, 0, 0.7)` (완전 어둡게 화면 전체를 차단)
2. **동기식 다운로드 수신:**
   - Axios의 응답이 올 때까지(Promise가 resolved 될 때까지) 로딩 레이어가 유지되므로, 사용자는 **다른 메뉴를 클릭하거나 다른 페이지로 이동할 수 없습니다.**
   - 응답이 완전히 오면 `downloadLoadingInstance.close()`를 호출해 로딩창을 닫고 `file-saver`를 통해 파일 저장을 완료합니다.

### 아키텍처적 평가 및 개선안
이 방식은 단시간에 끝나는 수천~수만 건의 데이터 다운로드에는 구현이 심플하고 에러 제어가 쉬워 유용하지만, **수십만 건을 다운로드하는 대용량 작업 시에는 사용성을 크게 저해하고 브라우저 연결 타임아웃**을 유발할 수 있습니다.

만약 대용량 엑셀 다운로드를 백그라운드에서 진행하면서 다른 화면 작업을 자유롭게 허용하려면 다음과 같은 **완전 비동기 배치식 다운로드 모델**로 개편해야 합니다.
1. **요청 즉시 즉시 반환:** 사용자가 다운로드 버튼을 누르면 서버는 즉시 파일 생성 작업의 `Job ID`만 반환하고 화면 차단을 풉니다.
2. **비동기 배치 생성 (Back-end):** 백엔드에서 `@Async`나 별도 스레드 풀(Spring Batch 등)을 구동해 엑셀 파일을 백그라운드에서 생성하고 파일 서버나 AWS S3에 업로드합니다.
3. **다운로드 센터 활용:** 사용자는 다른 업무를 하다가 "엑셀 다운로드 센터(가칭)" 메뉴로 이동하여 자신이 요청한 작업의 진행률(0% ~ 100%)을 확인하고, 완료되면 다운로드 링크를 눌러 받아가도록 구현합니다.
