현재 **L/T 프레임워크**의 엑셀 업로드/다운로드 핵심 로직인 `ExcelUtil.java` 및 `ExcelDrmUtil.java` 코드를 분석한 결과를 바탕으로 답변해 드립니다.

현재 프레임워크는 Java의 대표적인 엑셀 라이브러리인 **Apache POI**를 기반으로 작성되어 있습니다. **500개 열(Column) x 100만 개 행(Row)**(총 **5억 개의 셀**)의 초대용량 데이터를 처리할 때의 시간과 메모리 사용량은 다음과 같습니다.

### 1. 엑셀 파일 업로드 (Import)
* **메모리 요구량:** **100% OutOfMemoryError (OOM) 발생 (수백 GB 이상 필요)**
  * **원인:** `WorkbookFactory.create(is)`를 사용해 업로드된 파일을 읽습니다. 이 방식은 DOM 기반 파서(`XSSFWorkbook`)로, **엑셀 파일 전체 구조와 데이터를 JVM Heap 메모리에 Java 객체 형태로 전부 로드**합니다.
  * Apache POI 공식 문서 및 벤치마크에 따르면, `XSSFWorkbook`은 셀당 약 **0.5KB ~ 1.5KB**의 메모리를 차지합니다.
  * **5억 개 셀 × 0.5KB ~ 1.5KB = 최소 250GB ~ 750GB 이상의 JVM Heap Memory가 필요**합니다. 일반적인 WAS 환경에서는 **업로드 시작 즉시 서버가 다운(OOM)**됩니다.
  * 또한, 파싱된 데이터를 담을 `List<T>` 객체(100만 개 Row × 500개 Field)를 메모리에 유지하는 데만 추가로 약 **15GB ~ 30GB**의 Heap 메모리가 추가 필요합니다.
* **소요 시간:** **측정 불가 (메모리 부족으로 실패)**
  * 만약 하드웨어 스펙이 매우 높아서 수백 GB의 Heap 공간이 주어졌다고 가정하더라도, 5억 개의 셀을 메모리에 올리고 가비지 컬렉션(GC)을 수행하며 리플렉션(Reflection)을 통해 Java 객체로 매핑하는 작업은 **최소 30분에서 1시간 이상** 소요될 수 있습니다.
* **개선 제안:**
  * DOM 방식을 탈피하고 **SAX 방식의 스트리밍 파서**(`xlsx-streamer` 라이브러리 또는 Alibaba의 `EasyExcel`, `FastExcel` 등)를 적용해야 합니다.
  * 업로드 데이터를 한 번에 `List<T>`로 전부 받아오지 않고, 일정 단위(예: 1,000건)씩 **스트리밍 형식으로 읽어서 즉시 DB에 Bulk Insert하고 메모리에서 해제**하는 배치 아키텍처로 변경해야 OOM 없이 처리가 가능합니다.

### 2. 엑셀 파일 다운로드 (Export)
* **메모리 요구량:** **매우 높음 (최소 20GB ~ 35GB 이상의 JVM Heap 필요)**
  * **원인:** 현재 코드에서는 대용량 다운로드를 위해 임시 파일을 디스크에 쓰며 메모리를 절약하는 **`SXSSFWorkbook(500)`(Streaming XSSF)**을 사용하고 있습니다. SXSSF 자체는 지정된 500개 행의 메모리 윈도우만 유지하므로 엑셀 생성 자체의 메모리 점유율은 매우 낮습니다.
  * 하지만 `ExcelUtil`이 동작하기 전에 **100만 건의 데이터 전체(`List<T> list`)가 이미 메모리(JVM Heap)에 로드된 상태로 파라미터로 전달**되어야 합니다.
  * 500개의 필드를 가진 클래스 인스턴스 100만 개가 Heap 메모리에 상주하는 데 필요한 순수 객체 메모리만 약 **15GB ~ 30GB**입니다.
  * 따라서 WAS의 Heap 메모리가 최소 **32GB 이상**으로 넉넉하게 설정되어 있지 않다면, 데이터를 파일에 쓰기도 전에 DB 조회(`List` 반환) 단계 혹은 Excel 데이터 매핑 과정에서 **OOM이 발생**합니다.
* **소요 시간:** **약 5분 ~ 15분 내외** (서버 CPU 및 디스크 I/O 속도에 따라 상이)
  * JVM Heap 메모리가 충분하여 OOM을 피했다고 가정할 때, 5억 개의 셀을 스트리밍 파일로 쓰고 압축(Zip)하여 `.xlsx` 파일로 만들어 내보내는 데 걸리는 시간입니다.
  * 데이터 양이 막대하여 생성되는 최종 `.xlsx` 파일 크기 자체가 **500MB ~ 1.5GB**에 달할 수 있으므로, 최종 압축 및 다운로드 전송 속도에도 상당한 시간이 필요합니다.
* **개선 제안:**
  * DB 조회 시 `List` 전체를 메모리에 받지 않고, MyBatis의 `Cursor` 기능이나 JPA의 `Stream` API, 혹은 JDBC Fetch Size를 활용하여 **DB에서 데이터를 한 행씩 가져오면서 SXSSF로 바로 파일에 write하고, 이미 작성된 객체는 GC 대상이 되도록 청크 단위를 나누어야** 합니다.
  * 500개 열과 100만 행 규모의 다운로드는 일반 엑셀 다운로드 기능으로는 적합하지 않으므로, 백그라운드 배치 작업으로 수행한 뒤 사용자가 생성 완료 후 다운로드 링크를 통해 파일(zip/xlsx)을 직접 받게 하는 **비동기 배치 다운로드 방식**을 권장합니다.

결론부터 말씀드리면, **"현재 LT프레임워크의 구조로는 대용량 처리(예: 수십만 건 이상) 시 업로드와 다운로드 모두에서 OOM(OutOfMemoryError)이 발생할 가능성이 매우 높습니다."** 

일반적인 소량/중량 데이터(예: 수천 건~1만 건 수준)에서는 OOM 없이 안정적으로 작동하지만, 데이터가 비대해질수록 아키텍처 관점에서 치명적인 병목 지점들이 존재합니다. 구체적으로 분석한 원인은 다음과 같습니다.

### 🚨 1. 업로드 (Import) 시 OOM 원인
현 프레임워크의 `importExcelChunk` 메서드를 보면 이름은 `Chunk`이지만, 정작 엑셀 파싱의 첫 단추인 `WorkbookFactory.create(is)`에서 엑셀 파일의 모든 셀 데이터를 메모리에 로드하는 **DOM(Document Object Model) 방식**을 사용하고 있습니다.
* 데이터가 **10만~20만 건** 수준만 되어도 엑셀 객체들이 힙(Heap) 메모리를 가득 채우며 OOM을 일으키게 됩니다.
* 청크(Chunk) 단위로 쪼개어 가공하는 루프(`chunkSize = 10000`)를 구현해 두었으나, 이미 엑셀 파일 내용 전체가 JVM 메모리에 올라가 있는 상태이므로 이 루프는 OOM 방지에 아무런 실효성이 없습니다.

### 🚨 2. 다운로드 (Export) 시 OOM 원인
다운로드 시에는 임시 파일을 활용하는 `SXSSFWorkbook(500)` 스트리밍 방식을 채택했기 때문에 파일 생성 자체는 안전할 것 같지만, 두 가지 치명적인 문제가 있습니다.
1. **전체 데이터의 인메모리화:** 컨트롤러와 서비스 레이어에서 DB 조회를 수행할 때 `List<T> list` 형태로 100만 건(혹은 수십만 건)을 통째로 메모리에 로드하여 `exportExcel` 메서드로 넘깁니다. 100만 개의 자바 객체를 힙 메모리에 얹는 순간 이미 OOM 임계치에 도달합니다.
2. **스트리밍 플러시(Flush) 공식의 오류:** `writeSheetChunk` 코드를 보면, 플러시 단위를 호출하는 루프가 다음과 같이 작성되어 있습니다.
   ```java
   for (int i = start; i < end; i += list.size()) {
       int chunkEnd = Math.min(i + list.size(), end);
       fillExcelDataChunk(i, chunkEnd - i);
       ((SXSSFSheet) sheet).flushRows(list.size()); 
   }
   ```
   * 여기서 루프 증가폭이 `list.size()`(즉, 전체 데이터 크기인 100만)이기 때문에, 실질적으로 루프는 단 한 번만 돌며 **100만 건을 메모리에 다 쓰고 나서 마지막에 한꺼번에 플러시**하게 됩니다. 
   * 이렇게 되면 `SXSSFWorkbook`이 제공하는 "주기적 임시 디스크 플러시 및 메모리 해제" 메커니즘이 전혀 작동하지 않고, 일반 `XSSFWorkbook`처럼 작동하여 메모리가 급증하게 됩니다.

### 💡 요약 및 아키텍트 가이드
* **소/중용량(수만 건 이하):** 정상 작동함.
* **대용량(수십만 건 이상):** 업로드/다운로드 모두 높은 확률로 OOM 발생.
* **해결 방향:**
  * **업로드:** `WorkbookFactory.create(is)` 대신 `xlsx-streamer` 같은 스트리밍 라이브러리로 대체하고, 비즈니스 엔티티 생성 후 청크 단위로 즉시 DB에 반영 후 가비지 컬렉터가 수거하도록 유도해야 합니다.
  * **다운로드:** `List<T>`를 통째로 전달받는 구조를 MyBatis Cursor 등을 활용한 **스트리밍 DB 조회 & 엑셀 행 즉시 작성 및 주기적 플러시(`flushRows(1000)`)** 구조로 개편해야 합니다.
